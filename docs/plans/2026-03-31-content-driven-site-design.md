# Xthing.Link 内容驱动网站设计方案

**目标：** 将 Xthing.Link 明确定位为一个以技术写作为主入口的网站，让博客内容成为核心，项目与 Demo 作为论据、成果和延伸阅读存在。

**目标读者：** 喜欢动手折腾、关注工程权衡、软硬件实验、构建过程与踩坑记录的技术同好。

**当前约束：**
- 继续以 Astro 静态优先架构为基础。
- React islands 只在确实需要交互的页面或区块使用。
- 部署形态继续保持为静态 `dist/` 产物。
- 站点运行在阿里云轻量应用服务器上，可通过 Nginx 或 Caddy 提供静态站点服务。
- 现有 `blog` 和 `projects` 内容集合应扩展使用，而不是推倒重来。

---

## 1. 产品方向

Xthing.Link 不应做成传统意义上的作品集站点，而应做成一个技术写作主站。读者进入站点后，应当先看到你在思考什么、最近在研究什么，再顺着文章进入项目、Demo 和代码仓库。

网站应优先满足这些目标：
- 快速传达作者当前正在探索的主题。
- 让长期技术主题更容易被持续追踪。
- 将写作、项目和 Demo 组织成一个连贯的知识网络。
- 保持静态站点的低复杂度、低维护成本和良好 SEO。

这个方向强调的是深度、连续性和可发现性，而不是作品卡片式展示。

## 2. 信息架构

### 顶部导航

推荐的一级导航：
- `Home`
- `Writing`
- `Series`
- `Projects`
- `Demos`
- `About`
- `Search`
- `RSS`

### 导航设计理由

- `Home` 应该展示“最近在研究什么”，而不是只是栏目列表。
- `Writing` 比 `Blog` 更适合覆盖文章、笔记、构建记录、实验日志等内容形态。
- `Series` 值得单独成为一个导航入口，因为连续的专题内容比泛分类更能吸引技术读者。当前阶段已确认你能支撑 2 到 3 个专题，因此可以保留一级入口，但在首页中的视觉权重可以先低于 `Latest Writing`。
- `Projects` 仍然重要，但应当作为内容体系里的成果层和证据层。
- `Demos` 只承担交互展示，不承担完整叙事。
- `Search` 在内容逐渐增多后会成为高频入口，应该保留在一级结构中。

## 3. 首页结构

首页应从“站点目录页”升级为“内容驱动的首页”。

### 推荐区块

1. **Hero**
   - 用一句话说明你长期在折腾什么问题。
   - 可附一行补充，概括站点范围，例如嵌入式、可视化、个人工具、实验记录等。

2. **Latest Writing**
   - 展示最近 3 篇文章。
   - 每张卡片至少包含标题、简介、发布日期和标签。

3. **Ongoing Series**
   - 展示 2 到 3 个正在持续推进的专题。
   - 每个专题显示名称、简述、文章篇数和最近更新时间。
   - 在当前阶段，这个区块应存在，但视觉层级低于 `Latest Writing`，避免首页主轴被专题结构抢走。

4. **Featured Project**
   - 只展示 1 个重点项目，不要一开始就铺开很多项目卡片。
   - 应提供项目页入口和 Demo 入口。

5. **Notes / Lab Log**
   - 可作为后续扩展区块，用于承载较短的实验记录、阶段结论和碎片化思考。
   - 这一块可以等内容积累起来后再上。

### 首页应回答的问题

首页应在几秒钟内回答以下问题：
- 这个人最近在研究什么？
- 我现在最应该先读哪几篇内容？
- 哪个项目最能代表这些文章背后的实际工作？

## 4. 内容模型

当前的 Content Collections 结构是合理的，但为了支持筛选、互链和专题组织，还需要补充元数据。

### Blog 集合

当前字段：
- `title`
- `description`
- `pubDate`
- `updatedDate`
- `draft`

建议新增：
- `tags: string[]`
- `series?: string`
- `featured?: boolean`
- `hero?: string`
- `canonical?: string`
- `readingTime?: number`
- `summary?: string`
- `cover?: string`
- `platforms?: { name: string; url: string }[]`
- `source?: 'obsidian' | 'site'`
- `status?: 'draft' | 'published' | 'syndicated'`

### Blog 模型目的

- `tags` 用于标签页、搜索和主题筛选。
- `series` 用于把相关文章组织为长期专题。
- `featured` 用于首页或栏目页的精选推荐。
- `hero` 可为后续社交分享图或更强视觉表达预留。
- `readingTime` 在文章长度拉开后会有实际价值。
- `summary` 可用于站内列表、社交摘要和平台改写时的摘要提取。
- `cover` 为封面图、社交分享图和多平台适配预留。
- `platforms` 用于记录这篇文章已同步到哪些平台以及对应链接。
- `source` 用于明确内容是否来自 Obsidian 工作流。
- `status` 用于区分仅本地草稿、已站内发布、已同步分发等阶段。

### Projects 集合

当前字段：
- `title`
- `description`
- `status`
- `demoSlug`
- `order`

建议新增：
- `tags: string[]`
- `stack?: string[]`
- `repoUrl?: string`
- `demoUrl?: string`
- `startedAt?: date`
- `updatedAt?: date`
- `featured?: boolean`
- `relatedPosts?: string[]`

### Projects 模型目的

- `tags` 与 `stack` 支持按技术栈和主题浏览。
- `repoUrl` 与 `demoUrl` 为技术读者提供直接验证路径。
- `startedAt` 与 `updatedAt` 帮助展示项目时间线和活跃度。
- `featured` 用于精选项目展示。
- `relatedPosts` 用于将项目与相关文章显式关联。

## 5. 页面职责

### Home

作为作者当前研究方向的“总览页”，强调编辑感和选择性，而不是穷举所有内容。

### Writing

作为全站所有文章的总集合和主阅读入口，应同时支持按时间阅读、按标签进入、按专题进入。它不应只是纯标题列表。

### Series

作为 `Writing` 的专题组织层，而不是与 `Writing` 平行竞争的内容类型。也就是说，所有专题文章都属于 `Writing`，只有一部分文章会额外归入 `Series`。每个专题页建议说明：
- 为什么这个主题值得持续写
- 系列文章顺序
- 对应相关项目
- 当前推进状态

当前阶段你已经有 2 到 3 个可支撑的专题，因此 `Series` 可以保留在主导航中；但早期实现上要坚持“专题自然生长”的原则，避免为了凑结构而强行给每篇文章归类。

### Projects

作为“实践证据页”，每个项目页建议固定包含以下结构：
- 背景
- 目标
- 约束或关键难点
- 当前状态
- 技术栈
- 相关文章
- Demo 链接
- Repo 链接

项目页不应替代文章，而应承接文章。

### Demos

作为交互体验层，仅承载“看结果”和“玩交互”的职责，不应承担完整解释。Demo 页应能方便跳回项目页和相关文章。

### About

重点应放在你长期关注的问题域、工作方式、实验风格以及站点会产出什么内容，而不是只介绍技术栈。

### Search

目前“构建时嵌入索引 + 前端静态过滤”的方案是正确的。下一步优先做的是索引增强，而不是引入更复杂的搜索后端。建议纳入：
- tags
- series
- keywords
- project stack

## 6. 互链策略

这个站点长期价值的关键，在于让“思考过程”和“实现产物”形成互相支撑的网络。

建议建立这些互链规则：
- 每个项目页都应链接到相关的核心文章。
- 相关文章应能回链到对应项目页。
- 专题页可在适合时同时聚合文章和项目。
- Demo 页应至少能链接回对应项目页和一篇说明文章。

这样站点会更像一个知识图谱，而不是一堆彼此孤立的页面。

## 7. 内容生产与分发工作流

推荐采用“Obsidian 为唯一主稿源，网站为完整发布版本，外部平台为派生分发版本”的工作流。

### 工作流原则

- 在 Obsidian 中编写原始 Markdown 主稿。
- 网站优先发布最完整、最技术化、最适合长期沉淀的版本。
- 头条、知乎、小红书等平台不直接做一比一机械同步，而是从主稿派生出更适配平台语境的版本。
- 网站应尽量保留为“最完整原文”的主站。

### 推荐流程

1. 在 Obsidian 中完成文章初稿与修订。
2. 使用统一 frontmatter 进入站点内容目录。
3. 网站发布主稿，保留完整技术细节、代码、链接和上下文。
4. 针对外部平台生成派生稿：
   - 头条：适合更快进入观点和结论
   - 知乎：适合长文和完整论证
   - 小红书：适合图文拆解、实验过程和结果展示
5. 在站点文章 metadata 中记录各平台链接，用于内容管理和回流。

### 为什么不建议“全自动一键同步”

不同平台对标题、摘要、图片、语气和排版要求差异很大。过早追求完全自动化，通常会让维护成本上升，并牺牲内容质量。当前阶段更值得追求的是“一份主稿，多份轻量派生”，而不是“一个按钮发全平台”。

## 8. 编辑原则

为了保证网站随着内容增加仍然清晰，建议遵守以下原则：
- 当一个主题值得持续展开时，优先写成系列，而不是很多互不相连的单篇。
- 可以多发短而尖锐的技术笔记，不必等待所有文章都写成大而全的长文。
- 项目描述尽量具体、工程化，避免泛泛空话。
- 项目状态统一使用 `idea`、`wip`、`shipped` 这类稳定术语。
- 不要把项目页做成传统“作品集卡片墙”。

## 9. 建设优先级

建议的实施顺序：

1. 将首页改造成内容驱动首页。
2. 扩展 `blog` 与 `projects` 的 schema。
3. 建立 `Series` 体系和专题页。
4. 建立文章与项目的双向关联。
5. 最后再逐步充实 Demo 页。

## 10. 当前阶段非目标

- 不迁移到服务端渲染或重 API 架构。
- 不引入 CMS 作为前置条件。
- 不接入复杂的搜索后端。
- 不追求第一阶段就实现多平台全自动同步发布。
- 不在内容结构未稳定前，先大规模重做整套设计系统。

## 11. 设计实现衔接说明

当进入视觉设计和 UI 细化阶段时，再使用 `ui-ux-pro-max` 技能去参考和收敛设计风格。届时应选择更符合“技术、编辑感、内容导向”的风格，而不是通用型个人作品集视觉。
